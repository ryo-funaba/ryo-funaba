## 意欲・興味

常に目指していることは「**品質と製品価値を向上させて、ユーザーや社会から必要とされる製品を届けること**」です。

テストの 7 原則の 1 つに「バグゼロの落とし穴」が定義されているように、不具合が 1 つも存在しない完璧なシステムでもユーザーに使われない場合、実際には「品質が低い」と評価されます。

製品の価値を正しく届けるために、品質だけでなく**ユーザーの課題解決や顧客の目標達成**を第一に仕様策定や QA 活動に取り組みたいと考えています。

そのためには課題への深い理解、テスト技術、ステークホルダーとのコミュニケーションが特に重要と認識しています。

実際に、課題への理解を深めるためにビジネスドメインに関する資格や JSTQB というソフトウェアテストに関する資格を取得したので、**業務に必要な知識をインプットする習慣がある**ことは私の強みです。

今後の目標は「単に不具合を見つける人」ではなく「**プロダクトの価値を最大化する品質の担い手になること**」です。

#### 取り組んだ課題

##### テストケース数が多くスケジュール通りに終わらない

【課題】

本プロダクトの主要機能である商談の自動記録のシナリオは次の 4 つのステップに分かれています。

1. 顧客と日程調整
2. 顧客と商談を実施
3. 商談を自動記録
4. 自動記録された内容を CRM に連携

1、2、4 のステップごとに様々な日程調整の方法や外部連携サービス（Google、Zoom、Salesforce など）などの因子・水準があり、それらの組み合わせで問題なく動作するかテストする必要がありました。

しかし組み合わせのテストケースは多く、「1 ケースあたりのテスト実行時間 × テストケース数」をもとにテスト時間を見積もると、スケジュール内にテストを完了させることが難しい状況でした。

【解決策・工夫点】

テスト担当者の追加やスケジュールの調整が難しかったため、網羅的にテストを行うのではなく、次の作業によってテストケースを最適化して進める方針としました。

- 禁則（仕様上存在しない組み合わせ）のテストケースを洗い出し削減
- 組み合わせにより正常系の処理が動くかどうかを確認することが目的であったため、それ以外の観点（非機能検証など）を削り、検証観点を絞り込むことで 1 テストケースあたりのテスト実行時間を短縮
- 組み合わせが無則（組み合わせによって出力に変化がないこと）であることを開発者に確認し、目的がリグレッションテストだったため、ビジネス上の重要度や過去の不具合発生数を踏まえてリスクが高い特定の因子の組み合わせを優先
- 将来的に新機能追加により今回作成したテストを再実行されることが見込まれていたため、テストに必要なデータやアカウント情報、テストケースを整理し自分以外のチームメンバーが再実行できるよう整備

【成果】

全ての組み合わせのテストケースをリスクベースで最適化したことで、テストケースを 80% 削減しスケジュール内にテストを完了させました。
その後、テストした内容に関するデグレードは報告されず、品質を維持したままのテスト効率化に成功しました。

また、新機能追加により今回作成したテストが再実行される際、テストに必要な情報が揃っており自分以外のメンバーがテストを実行できたため、テストの生産性向上や属人性の解消についても寄与しました。

上記の取り組みを通して、必要なテストケースを洗い出すだけでなく、不要なテストケースを削減するスキルも重要であると認識しました。

#### 取り組んだ課題

##### 開発時に必須となるドキュメントが引き継がれなかった

【課題】

他社が開発したシステムを引き継ぎ、自社で保守・運用や新機能開発を担当する方針でプロジェクトがスタートしました。

しかし引き継ぎにおいて、開発に必要な API や DB の設計書、環境構築の手順書、インフラ構成図、リリース手順書はほとんど存在せず、コードと本番環境のアカウント情報のみが引き継がれた状況でした。

以上の状況によりプロジェクトを安定的に進めることが困難な状況でした。

【解決策・工夫点】

ユーザーからの問い合わせの調査や不具合の修正と並行しながら、プロジェクトの継続・拡大に向けて開発基盤の整備に取り組みました。
具体的な内容は次の 4 つです。

1 つ目はローカルの環境構築の改善です。

全チームメンバーが同じ環境で迅速に開発できる体制を実現するために Docker を新たに導入して環境をコンテナ化しました。

Docker を導入する際は、キャッシュの活用やマルチステージビルドなど公式のベストプラクティスに沿って実装しました。

その他環境構築に必要なライブラリやコマンドは 1 つの make コマンドで実行できるよう改善しました。

2 つ目はコーディング規約の定義です。

引き継ぎ当初はコーディング方針が統一されておらず、可読性や保守性が低い状態でした。

そこで Go 公式や Uber のコーディング規約を参考に、プロジェクトの規模やチームメンバーのレベルに合わせたコーディング規約を定義しました。

具体的にはエラーハンドリング、ログ出力、クリーンアーキテクチャの各層の責務などを定義しました。

3 つ目は CI の整備です。

手動テストやレビューなど人力によって品質を担保していため、開発効率を上げるために Lint と Go のユニットテストを導入しました。

ユニットテストでは RDB や AWS（DynamoDB、SES、SQS など）の外部サービスとの通信に関するロジックをテストするよう実装しました。

AWS などの CI 上で環境を再現できないテストケースではモックを活用しました。

4 つ目はドキュメントの作成です。

API や DB の設計書、環境構築の手順書、インフラ構成図、リリース手順書を作成しました。

また、技術構成がチームメンバーにとって馴染みのない GraphQL とサーバーレスアーキテクチャ（Lambda + DynamoDB）だったため、それらの技術的特徴やリクエストが処理されるフローについて解説したドキュメントも作成しました。

【成果】

1 つ目のローカルの環境構築の改善により、2 時間を要していた環境構築がコマンド 1 つで 5 分で完了できるよう改善されました。

これにより軽微な修正の PR であれば新規メンバーがアサインされた当日に作成できる状態となりました。

2 つ目のコーディング規約の定義により、レビューでのコーディングスタイルに関する指摘が減少し、設計やロジックについて議論する時間が増えました。

3 つ目の CI の整備により継続的にコードの品質が担保される仕組みとなりました。

レビュー前にロジックやコーディング規約に関して自動で検証されることで、レビューでの指摘事項が減りレビュー効率を改善しました。

モックを用意したことでローカルでの開発時に AWS のリソースにアクセスせずとも安全・効率的にロジックを検証できるようになりました。

4 つ目のドキュメントの作成により誰もがアーキテクチャや仕様を理解できる状態となり、オンボーディングも改善されたため環境構築を含めて新規メンバーがアサインされた当日に PR を作成できる状態となりました。

引き継ぎ資料には存在しなかったバッチやリリースの手順書もドキュメント化し、私以外のメンバーがバッチやリリースを実施できる体制となり属人性の防止も実現しました。

上記の開発体制の強化を通して、開発メンバーが当初の 2 人から 6 人まで増員されプロジェクトの継続・拡大が実現しました。

また増員による新機能開発のスピード向上やユーザー数の増加により、運営企業の**東証グロース市場への上場**に貢献しました。

これらの成果が評価され、社内の全マネージャーの投票の結果、**最優秀新人賞**を受賞しました。
