品質とユーザー価値の最大化を重視している。

## 意欲・興味

常に目指していることは「**品質と製品価値を向上させて、ユーザーや社会から必要とされる製品を届けること**」です。

テストの 7 原則の 1 つに「バグゼロの落とし穴」が定義されているように、不具合が 1 つも存在しない完全無欠なシステムでもユーザーに使われない場合、実際には「品質が低い」と評価されます。

製品の価値を正しく届けるために、品質だけでなく**ユーザーの課題解決や顧客の目標達成**を第一に QA 活動を進めたいと考えています。

そのためには課題への深い理解、テスト技術、ステークホルダーとのコミュニケーションが特に重要と認識しています。

実際に、課題への理解を深めるためにビジネスドメインに関する資格やソフトウェアテストに関する資格を取得したので、**業務に必要な知識をインプットする習慣がある**ことは私の強みです。

今後の目標は「単に不具合を見つける人」ではなく「**プロダクトの価値を最大化する品質の担い手になること**」です。

#### プロダクト概要

2025 年 2 月以降は QA エンジニアとしてテストの設計〜実行、仕様のレビュー、E2E テストの導入、不具合調査を担当しました。

受け身でプロダクトマネージャーやデザイナーからの要望をそのまま実現するのではなく、限られた工数の中で目的達成のために最善の仕様やデザインの改善案を提案しました。

#### 取り組んだ課題

##### ①手動テストの工数が多く、リリースまでに時間がかかる

**【課題】**

1 つの機能に対して複数の画面やボタンからアクセスできる設計となっており、軽微な修正の場合でも様々な画面やボタンから手動テストしなければならず、テスト工数の増加やテストの抜け漏れによるデグレードが発生していました。

**【解決策・工夫点】**

E2E テストを導入し、手動テストの工数が多い機能や過去にデグレードが発生した機能から優先的に自動テストされるよう整備しました。

テストツール選定では「並列実行」「標準機能による自動待機」「モジュール化による高い保守性」を基準として検討し Playwright を採用しました。

高い保守性を実現するために、アーキテクチャは Screen Action Pattern としました。
具体的には UI 要素、操作（入力やボタン押下などの操作とアサーション）、シナリオ（複数の操作を組み合わせた一連のシナリオ）の 3 クラスに分離する設計としました。

E2E テストの結果とエラーの詳細は GitHub Actions の Job Summaries 機能を活用し、ジョブの詳細画面からすぐに確認できる状態にしました。

利用中のアナリティクスツールのポップアップの表示により E2E テストが不安定になるため、カスタムフィクスチャを導入してアナリティクスツールへのリクエストをブロックするよう改善しました。

**【成果】**
<!-- E2E テストを導入したことで、手動テストの工数が -->
Screen Action Pattern を採用したことで UI やロジックなどの変更の影響範囲が限定され、モジュール化によるコードの再利用性が向上しました。
各クラスの責務が明確に分離されることで Page Object Pattern のケースと比較して **AI によるテストケース作成の精度が向上**しました。

Job Summaries 機能を活用したことで、エラーの詳細を確認するために視認性が低い大量のログの中から探すという手間がなくなりました。
Slack に通知されたテスト結果画面の URL に遷移するだけで、**エラーの詳細を誰でも簡単に確認できる**ようになりました。

アナリティクスツールへのリクエストをブロックしたことで安定的に E2E テストが実行できるようになり、テストの信頼性が向上しました。

##### ②テストケース数が多くスケジュール通りに終わらない

**【課題】**

主要機能では様々な外部サービス（Google、Zoom、Salesforce など）と連携しており、新機能追加後にそれらの組み合わせで問題なく動作するかリグレッションテストする機会がありました。

しかし組み合わせのテストケースは多く、「1 ケースあたりのテスト実施時間 × テストケース数」をもとに実施時間を見積もると、スケジュール内に完了させることが難しい状況でした。

**【解決策・工夫点】**

テスト担当者の追加やスケジュールの調整が難しかったため、網羅的にテストを行うのではなく、次の作業によってテストケースを最適化して進める方針としました。

- 禁則（仕様上存在しない組み合わせ）のテストケースを洗い出し削減
- 検証観点を正常系の観点のみに絞り込み 1 テストケースあたりのテスト実施時間を短縮
- 組み合わせが無則（組み合わせによって出力に変化がないこと）であることを開発者に確認し、ビジネス上の重要度や過去の不具合発生数を踏まえてリスクが高い特定の因子の組み合わせを優先

**【成果】**

全ての組み合わせのテストケースをリスクベースで最適化したことで、**テストケースを 80% 削減**しスケジュール内にテストを完了させました。
リリース後、テストした内容に関するデグレードは報告されず、品質を維持したままのテスト効率化に成功しました。

その後本テストが再実施される際に必要なデータやアカウント情報が揃っており、他のメンバーでもすぐにテストを実施できたため、テストの生産性向上や属人化の防止についても寄与しました。

上記の取り組みを通して、必要なテストケースを洗い出すだけでなく、不要なテストケースを削減するスキルも重要であると認識しました。

#### 取り組んだ課題

##### 開発時に必須となるドキュメントが引き継がれなかった

**【課題】**

他社が開発したシステムを引き継ぎ、プロジェクトがスタートしました。
しかし開発に必要なドキュメントはほとんど存在せず、コードと本番環境のアカウント情報のみが引き継がれた状況でした。

以上の状況によりプロジェクトを安定的に進めることが困難な状況でした。

**【解決策・工夫点】**

ユーザーからの問い合わせの調査や不具合の修正と並行しながら、プロジェクトの継続・拡大に向けて開発基盤の整備に取り組みました。
具体的な内容は次の 4 つです。

①ローカルの環境構築の改善

全チームメンバーが同じ環境で迅速に開発できる体制を実現するために Docker を導入し、コマンド 1 つで環境構築できるよう改善しました。

②コーディング規約の定義

引き継ぎ当初はコーディング方針が統一されておらず、可読性や保守性が低い状態でした。
そこで Go 公式や Uber のコーディング規約を参考にエラーハンドリング、ログ出力、クリーンアーキテクチャの各層の責務などのコーディング規約を定義しました。

③ CI の整備

手動テストやレビューなど人力によって品質を担保していため、開発効率を上げるために Lint と Go のユニットテストを導入しました。
AWS などの CI 上で環境を再現できないテストではモックを活用しました。

④ドキュメントの作成

API や DB の設計書、環境構築の手順書、インフラ構成図、リリース手順書を作成しました。
また、会社やチームメンバーにとって馴染みのない技術構成だったため、それらの特徴や処理の流れについて解説したドキュメントも用意しました。

**【成果】**

①のローカルの環境構築の改善により、2 時間を要していた環境構築がコマンド 1 つで 5 分で完了できるよう改善されました。

②のコーディング規約の定義や③の CI の整備により、レビューでのコーディングスタイルに関する指摘が減少し、設計やロジックについて議論する時間が増えました。
モックを用意したことでローカルでの開発時に AWS のリソースにアクセスせずとも安全・効率的にロジックを検証できるようになりました。

④のドキュメントの作成により、新規メンバーのキャッチアップをサポートし、オンボーディングプロセスが改善されました。
環境構築を含めて新規メンバーがアサインされた当日に PR を作成できる状態となりました。
バッチやリリースの手順書もドキュメント化したことで、私以外のメンバーが対応できる体制となり属人化の防止に寄与しました。

上記の開発体制の強化を通して、開発メンバーが当初の 2 人から 6 人まで増員されプロジェクトの継続・拡大が実現しました。
増員による新機能開発のスピード向上やユーザー数の増加により、運営企業の**東証グロース市場への上場**に貢献しました。

これらの成果が評価され、社内の全マネージャーの投票の結果、**最優秀新人賞を受賞**しました。
